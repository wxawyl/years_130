
# Live to 130 需求影响分析报告

## 需求概述

本次分析包含四个用户需求，其中需求1和需求2为重复内容（个人健康知识库RAG），需求3为优雅视觉交互，需求4为主动式互动。

### 需求列表

| 编号 | 需求名称 | 优先级 |
|------|---------|--------|
| 1-2 | 个人健康知识库（RAG） | P0 |
| 3 | 优雅视觉交互 | P1 |
| 4 | 主动式互动 | P1 |

## 一、需求1-2：个人健康知识库（RAG）

### 1.1 需求描述
利用本地向量数据库存储用户过去一年的所有行为数据，支持自然语言查询，如"我上个季度身体状态最好的一周是怎么过的？"，能够瞬间调取数据给出精准画像。

### 1.2 当前系统状态

#### 现有数据存储
```
lib/services/database_service.dart
- MemoryDatabase: 内存存储，使用List&lt;Map&gt;结构
- 支持的数据表: sleep_records, diet_records, exercise_records, voice_diaries, daily_scores等
- 问题：内存存储无持久化，查询能力有限
```

#### 现有知识库服务
```
lib/services/knowledge_service.dart
- 静态健康知识内容
- 按类别组织（睡眠、饮食、运动、情绪）
- 问题：仅静态内容，无用户个性化数据检索
```

### 1.3 影响范围分析

#### 1.3.1 数据层影响 🔴 高

**需要新增：**
- 向量数据库集成（可选方案：sqflite + 自定义向量索引 / 或使用 SQLite FTS5 扩展）
- 数据向量化服务
- 向量相似度搜索算法
- 数据持久化方案（替代当前的内存存储）

**现有修改：**
- `database_service.dart` - 需要完全重构以支持向量存储
- 所有数据模型需要增加向量嵌入字段

#### 1.3.2 服务层影响 🔴 高

**新增服务：**
```
lib/services/
├── vector_database_service.dart    # 向量数据库操作
├── embedding_service.dart          # 文本向量化
├── rag_query_service.dart          # RAG查询处理
├── health_profile_service.dart     # 健康画像生成
└── knowledge_base_service.dart     # 知识库管理（重构）
```

**修改服务：**
- `ai_integration_service.dart` - 集成RAG能力
- `insight_discovery_service.dart` - 增强发现能力

#### 1.3.3 前端层影响 🟡 中

**新增页面：**
```
lib/screens/
├── knowledge_query_screen.dart     # 知识库查询页面
├── health_profile_screen.dart      # 健康画像展示页面
└── knowledge_management_screen.dart # 知识库管理页面
```

**修改页面：**
- `home_screen.dart` - 增加知识库入口
- 各功能页面 - 增加数据同步到知识库的逻辑

#### 1.3.4 原生层影响 🟡 中

- **iOS**: 可能需要Core ML集成来运行本地嵌入模型
- **Android**: 可能需要TensorFlow Lite集成
- **Web**: 降级方案使用云端API或WASM

### 1.4 技术方案建议

#### 方案A：轻量级本地向量存储（推荐）
```yaml
依赖:
  - sqflite: 现有依赖，可扩展
  - 自定义向量索引: 基于余弦相似度
  - 本地嵌入模型: 使用量化的small BERT模型
优点: 完全离线，隐私保护，轻量级
缺点: 需要更多开发工作
```

#### 方案B：混合方案
```yaml
本地存储: 结构化数据（sqflite）
向量化: 可选本地/云端（用户可选择）
优点: 灵活性高，可降级
缺点: 复杂度增加
```

### 1.5 数据结构设计

#### 向量记录表
```dart
class VectorRecord {

  final String originalDataId;
  final List&lt;double&gt; embedding; // 向量嵌入
  final String metadata; // JSON格式的元数据
  final DateTime timestamp;
  final String contentSummary; // 文本摘要用于向量化

```

#### 健康画像数据结构
```dart
class HealthProfile {
  final DateTime startDate;
  final DateTime endDate;
  final Map&lt;String, dynamic&gt; metrics; // 各项指标
```

```dart
// 动态主题应用
class DynamicThemeContainer extends StatelessWidget {

  }
```

### 3.3 主动式互动场景分析

#### 3.3.1 健康异常类

| 场景 | 触发条件 | 互动方式 | 优先级 |
|------|---------|---------|--------|
| 心率异常 | 静息心率 &gt; 100 或 &lt; 50 | 温和询问 + 建议 | P0 |
| 睡眠不足 | 连续3天睡眠 &lt; 6小时 | 关怀提醒 + 睡眠建议 | P0 |
| 久坐不动 | 连续8小时无运动记录 | 轻微提醒 + 活动建议 | P1 |
| 情绪低落 | 连续负面情绪记录 | 温暖问候 + 调节建议 | P1 |
| 饮食不规律 | 错过2餐以上记录 | 温和提醒 | P1 |

#### 3.3.2 积极强化类

| 场景 | 触发条件 | 互动方式 | 优先级 |
|------|---------|---------|--------|
| 连续达标 | 连续7天健康评分 &gt; 80 | 庆祝动画 + 小成就 | P1 |
| 习惯养成 | 坚持某项运动21天 | 成就解锁 + 鼓励 | P1 |
| 改善进步 | 某项指标显著改善 | 正向反馈 + 建议保持 | P2 |

#### 3.3.3 环境关联类

| 场景 | 触发条件 | 互动方式 | 优先级 |
|------|---------|---------|--------|
| 天气变化 | 温差大/空气质量差 | 健康提醒（注意保暖/戴口罩） | P2 |
| 季节变化 | 换季时期 | 季节性健康建议 | P2 |

## 六、总结与建议

### 6.1 核心结论

1. **RAG知识库**是最具技术挑战性但也最有价值的需求，将构建产品核心竞争力
2. **视觉交互**相对独立，能快速提升用户体验，建议优先实施
3. **主动式互动**需要前两者的基础，但能显著提升产品粘性
4. 三个需求相辅相成，建议按优先级分阶段实施

### 6.2 实施建议

✅ **推荐策略：**
- 先做**视觉交互**（快速见效，用户感知强）
- 同时进行**数据持久化改造**（为RAG打基础）
- 再做**RAG知识库**核心功能
- 最后完善**主动式互动**

⚠️ **注意事项：**
- 保持隐私优先原则，所有功能设计考虑本地处理
- 提供用户完全的控制权（可关闭任何功能）
- 分阶段灰度发布，收集用户反馈
- 保持代码质量，避免过度设计

### 6.3 后续展望

这三个需求的实施将使产品从"数据记录工具"升级为"智能健康伴侣"，具备：
- 🧠 记忆能力（RAG知识库）
- 🎨 共情能力（视觉交互）
- 💝 关怀能力（主动式互动）

这将构建显著的产品差异化优势。
