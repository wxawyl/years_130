# Project Rules - Live to 130

## 核心开发规则

### 1. 标准开发流程（必须遵循）

**规则：所有开发活动必须按照以下标准流程执行，每个阶段使用对应的技能。**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    标准开发流程（SDLC）                              │
├─────────────────────────────────────────────────────────────────────┤
│  阶段1: 需求分析    →  requirements-analyst                         │
│       ↓                                                             │
│  阶段2: 影响评估    →  requirement-impact-analyst                   │
│       ↓                                                             │
│  阶段3: UI/UX设计   →  ui-ux-pro-max / frontend-design              │
│       ↓                                                             │
│  阶段4: 开发实现    →  fullstack-developer                          │
│       ↓                                                             │
│  阶段5: 测试验证    →  webapp-testing                               │
│       ↓                                                             │
│  阶段6: Bug修复     →  bug-fixer (循环迭代)                          │
└─────────────────────────────────────────────────────────────────────┘
```

#### 各阶段详细说明

| 阶段 | 技能 | 职责 | 输出 |
|------|------|------|------|
| **阶段1: 需求分析** | `requirements-analyst` | 分析需求、识别差距、文档化 | 需求文档、用户故事 |
| **阶段2: 影响评估** | `requirement-impact-analyst` | 评估影响范围、识别风险 | 影响分析报告 |
| **阶段3: UI/UX设计** | `ui-ux-pro-max` / `frontend-design` | 设计界面、交互原型 | 设计稿、组件规范 |
| **阶段4: 开发实现** | `fullstack-developer` | 实现功能、集成测试 | 代码、单元测试 |
| **阶段5: 测试验证** | `webapp-testing` | 自动化测试、端到端测试 | 测试报告、Bug清单 |
| **阶段6: Bug修复** | `bug-fixer` | 定位bug、修复验证 | 修复代码、回归测试 |

#### 触发条件

| 场景 | 适用流程 |
|------|---------|
| **新功能开发** | 完整流程（阶段1-6） |
| **功能修改/优化** | 阶段2-6 |
| **Bug修复** | 阶段6（必要时阶段2） |
| **设计优化** | 阶段3-5 |

#### 阶段2：影响评估内容必须包括
- 影响范围（前端、后端、数据层、第三方依赖）
- 风险评估（风险类型、风险等级、缓解策略）
- 实现建议（优先级、技术方案、时间预估）

---

### 2. 产品设计理念

#### 大众共创理念
- 用户反馈入口设计突出，体现尊重用户想法的意识
- 分享社区让用户可以分享自己的健康智慧
- 支持多语言内容，体现国际化视野

#### 开源协作理念
- 插件市场：允许开发者创建和分享插件
- API开放：提供标准化数据接口
- AI模型市场：分享和发现健康分析模型
- 贡献者激励：徽章、排行榜、荣誉墙

---

### 3. 国际化要求

- 所有新功能必须支持多语言
- 支持的语言：中文、英文、日语、韩语、西班牙语
- 考虑跨文化适配和地区差异

---

### 4. 代码规范

- 遵循 Flutter 最佳实践
- 使用现有的服务模式（*_service.dart）
- 保持代码风格一致性
- 添加必要的错误处理

---

### 5. 测试要求

- 完成开发后运行 lint 和 typecheck
- 确保不引入新的 bug
- 验证多语言支持
