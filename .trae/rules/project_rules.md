# Project Rules - Live to 130

## 核心开发规则

### 0. 隐私优先原则（最高优先级）

**规则：所有功能设计必须遵循隐私优先原则，以本地处理为首选方案。**

- **绝对隐私承诺**：明确向用户保证"情绪私语、体重数据、饮食习惯全部经过加密存储在本地，AI推理在手机端芯片完成"
- **本地优先**：所有计算优先考虑在设备本地完成，仅在必要时使用云端服务
- **数据加密**：所有用户敏感数据必须使用 AES-256-GCM 加密存储
- **透明性**：提供隐私保护展示页面，清晰说明数据处理方式
- **可选云端**：云端服务作为可选降级方案，用户可自主选择

#### 本地 AI 开发规范

| 方面 | 要求 |
|------|------|
| **架构设计** | 支持云端/本地/混合三种模式切换 |
| **模型选择** | 优先使用轻量级量化模型（Phi-3 Mini, TinyBERT 等） |
| **平台支持** | iOS (Core ML) → Android (TensorFlow Lite) → Web (WASM/降级) |
| **加密标准** | AES-256-GCM 数据加密，Keychain/Keystore 密钥管理 |

### 1. 标准开发流程（必须遵循）

**规则：所有开发活动必须按照以下标准流程执行，每个阶段使用对应的技能。**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    标准开发流程（SDLC）                              │
├─────────────────────────────────────────────────────────────────────┤
│  阶段1: 需求分析    →  requirements-analyst                         │
│       ↓                                                             │
│  阶段2: 影响评估    →  requirement-impact-analyst                   │
│       ↓                                                             │
│  阶段3: UI/UX设计   →  ui-ux-pro-max / frontend-design              │
│       ↓                                                             │
│  阶段4: 开发实现    →  fullstack-developer                          │
│       ↓                                                             │
│  阶段5: 测试验证    →  webapp-testing                               │
│       ↓                                                             │
│  阶段6: Bug修复     →  bug-fixer (循环迭代)                          │
└─────────────────────────────────────────────────────────────────────┘
```

#### 各阶段详细说明

| 阶段 | 技能 | 职责 | 输出 |
|------|------|------|------|
| **阶段1: 需求分析** | `requirements-analyst` | 分析需求、识别差距、文档化 | 需求文档、用户故事 |
| **阶段2: 影响评估** | `requirement-impact-analyst` | 评估影响范围、识别风险 | 影响分析报告 |
| **阶段3: UI/UX设计** | `ui-ux-pro-max` / `frontend-design` | 设计界面、交互原型 | 设计稿、组件规范 |
| **阶段4: 开发实现** | `fullstack-developer` | 实现功能、集成测试 | 代码、单元测试 |
| **阶段5: 测试验证** | `webapp-testing` | 自动化测试、端到端测试 | 测试报告、Bug清单 |
| **阶段6: Bug修复** | `bug-fixer` | 定位bug、修复验证 | 修复代码、回归测试 |

#### 触发条件

| 场景 | 适用流程 |
|------|---------|
| **新功能开发** | 完整流程（阶段1-6） |
| **功能修改/优化** | 阶段2-6 |
| **Bug修复** | 阶段6（必要时阶段2） |
| **设计优化** | 阶段3-5 |

### 1.5 自动化影响评估（默认启用）

**规则：所有开发活动，无论大小，默认自动调用 Requirement Impact Analyst 进行影响评估。**

**自动触发场景：**
- ✅ 任何新功能开发请求
- ✅ 功能修改或优化请求
- ✅ 用户体验改进请求
- ✅ 架构调整请求
- ✅ 任何用户明确提出的需求变更

**自动流程：**
1. 用户提出需求 → 自动调用 Requirement Impact Analyst
2. 生成影响评估报告 → 确认后进入开发流程
3. 开发完成 → 验证与测试

**无需手动声明：**
用户无需每次都明确说"使用 Requirement Impact Analyst"，系统会自动应用此规则。

#### 阶段2：影响评估内容必须包括
- 影响范围（前端、后端、数据层、第三方依赖）
- 风险评估（风险类型、风险等级、缓解策略）
- 实现建议（优先级、技术方案、时间预估）

### 2. 产品设计理念

#### 大众共创理念
- 用户反馈入口设计突出，体现尊重用户想法的意识
- 分享社区让用户可以分享自己的健康智慧
- 支持多语言内容，体现国际化视野

#### 开源协作理念
- 插件市场：允许开发者创建和分享插件
- API开放：提供标准化数据接口
- AI模型市场：分享和发现健康分析模型
- 贡献者激励：徽章、排行榜、荣誉墙

## 本地 AI 技术栈

| 平台 | 技术方案 | 推荐模型 |
|------|---------|---------|
| **iOS** | Core ML + Metal | Phi-3 Mini (4-bit), TinyBERT |

| **Web** | WASM + 云端降级 | 降级使用云端 API |

## 模型管理规范

1. **模型文件**：使用 `.gguf` 量化格式
2. **存储位置**：`/models/` 应用文档目录
3. **下载管理**：支持断点续传、进度显示
4. **版本管理**：支持模型版本更新和回滚
